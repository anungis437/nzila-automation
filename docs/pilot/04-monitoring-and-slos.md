# Monitoring & SLOs for Pilots

**Purpose:** Explain how Nzila OS monitors pilot deployments and what SLO guarantees are in place.
**Owner:** Platform Engineering / Ops
**Last Updated:** 2026-03-03

---

## 1. SLO Summary

All pilots operate under the Nzila OS SLO policy (`ops/slo-policy.yml`):

| Metric | Default Threshold | Enforcement |
|--------|-------------------|-------------|
| P95 Latency | ≤ 500ms | Deploy blocked if exceeded in pilot/prod |
| P99 Latency | ≤ 2000ms | Deploy blocked if exceeded in pilot/prod |
| Error Rate | ≤ 2.0% | Deploy blocked if exceeded in pilot/prod |
| Integration Success Rate | ≥ 99.0% | Deploy blocked if exceeded in pilot/prod |
| Integration Delivery P95 | ≤ 5000ms | Deploy blocked if exceeded in pilot/prod |
| DLQ Backlog | ≤ 100 events | Deploy blocked if exceeded in pilot/prod |

> Per-app overrides are available — see `ops/slo-policy.yml` for app-specific thresholds.

## 4. Automated Alerts

### Daily Health Digest

A daily health digest is generated by `@nzila/platform-ops/health-digest`:
- Computes deltas for p95 latency, error rate, integration SLA, DLQ backlog.
- Writes snapshot to the digest store.
- Emits audit event: `platform.ops.health_digest.generated`.
- Sends summary to Slack / Teams via ChatOps adapters.

### Anomaly Alerts

When a metric breaches the SLO threshold:
- **Warning:** Value exceeds SLO → digest summary includes warning.
- **Critical:** Value exceeds 1.5× SLO → immediate Slack/Teams alert with Console link.

### Alert Routing

| Channel | Target | Content |
|---------|--------|---------|
| Slack `#ops-alerts` | Ops team | All alerts + daily digest |
| Teams `Ops Alerts` | Ops team | All alerts + daily digest |
| PagerDuty | On-call | P1/P2 incidents only |

---

## 5. SLO Gate in CI/CD

Every deploy to pilot or prod runs through the SLO gate (`scripts/slo-gate.ts`):

1. Reads current metrics from platform stores.
2. Compares against `ops/slo-policy.yml` thresholds.
3. **If violations detected → deploy blocked.**
4. Generates release attestation (`scripts/release-attestation.ts`).

This ensures no regression ships to a customer environment.

---

## 6. What Happens When an SLO Is Breached?

1. Alert fires to on-call via ChatOps + PagerDuty.
2. On-call follows the appropriate runbook (see [runbooks](../ops/runbooks/)).
3. Incident ticket created if SLA is at risk.
4. Postmortem for P1/P2 incidents within 5–10 business days.
5. All actions audited and evidence-sealed.
