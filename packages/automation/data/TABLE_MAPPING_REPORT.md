## Part 3: Cross-Platform Observations

### 3.1 Shared Table Names (Same `db_table` on Both Platforms)

The following table names appear in BOTH UE and ABR Django models. They use the **same** `db_table` value but may have different column sets:

| Table Name | UE App | ABR App | Notes |
|------------|--------|---------|-------|
| `organizations` | auth_core | auth_core | **Shared model** — merged UE+ABR schema in Django |
| `organization_relationships` | auth_core | auth_core | Shared model |
| `profiles` | auth_core | auth_core | Different column sets (UE has membership fields) |
| `sso_providers` | auth_core | core | Different apps, similar concept |
| `stripe_webhook_events` | billing | core | Different apps |

### 3.2 Column Rename Patterns

| Pattern | Description | Examples |
|---------|-------------|----------|
| **FK convention** | Django FK field `organization` → column `organization_id` | All models with `organization` FK |
| **Python reserved words** | Source SQL column name is a Python keyword | `not` → `not_field` (quiz_questions), `or` → `or_field` (resource_permissions), `like` → `like` (audit_logs_archive — risky) |
| **Dropped columns** | Source columns not included in Django model | See per-table notes (e.g., `search_vector`, `embedding`) |

### 3.3 Skip Columns (Present in Source, Absent in Django)

| Column Type | Reason | Affected Tables |
|-------------|--------|-----------------|
| `search_vector TSVECTOR GENERATED` | PostgreSQL-specific generated column; rebuild via Django SearchVector | `courses`, `lessons`, `tribunal_cases`, `organization_members`, `knowledge_base` |
| `embedding vector(1536)` | pgvector column; requires VectorField (conditionally imported) | `case_embeddings`, `course_embeddings`, `lesson_embeddings`, `knowledge_base` |
| Supabase RLS policies | Not columns, but access policies | All ABR tables |
| `auth.users` FK | Supabase-specific auth reference | `profiles.id` in ABR (references `auth.users(id)`) |

## Part 4: Migration Data Transfer Summary

### Rules for All Tables

1. **Primary Key**: Source UUID `id` → Django auto-generated PK (configure UUID PK in models if needed)
2. **Timestamps**: `created_at` / `updated_at` → preserve original values, don't use auto_now during migration
3. **JSONB columns**: Direct transfer — Django JSONField handles these natively
4. **ArrayField columns**: Direct transfer — Django ArrayField (postgres-specific)
5. **Enum columns**: Source CHECK constraints → Django CharField with choices
6. **Soft deletes**: `deleted_at` columns → preserve as-is
7. **TSVECTOR columns**: Skip during migration; rebuild using Django `SearchVector` after data load
8. **pgvector columns**: Transfer if VectorField is available; skip otherwise

### Migration Order (Topological Sort by FK Dependencies)

**UE**: Organizations → Profiles/Users → OrganizationMembers → Employers/Worksites/BargainingUnits → all child tables

**ABR**: Organizations → Profiles → Roles/Permissions → RolePermissions/UserRoles → Courses/Lessons → Enrollments → all child tables
